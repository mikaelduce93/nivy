# Economy — Vision audit

> Generated by `economy-vision-auditor` on 2026-05-07. Read-only audit of the
> 2-currency economic model crossing founder vision, code, and live DB state
> (Supabase project `imchornjvmgmaovhypco` — "nivy"). All file paths absolute,
> rooted at `C:\Users\Shadow\Desktop\NIVY`.

## 1. Vision intended

Nivy is built around two complementary currencies for Moroccan teens 13-17 and
their parents:

- **XP** — earned by the teen via challenges, quizzes, daily streaks, group
  activities, fortune wheel, missions. It is the engagement signal and the
  primary "I worked for it" token.
- **Coins** — purchased by the parent in dirhams (DH) through top-up packs and
  credited to the teen's wallet. Coins are the "real money proxy" the parent
  controls; they fund partner discounts, event tickets, mystery boxes and
  cosmetic profile customisation.

The founder doc `docs/economy.md` (read first) actually documents **three**
in-app constructs: (1) `user_xp.total_xp`, (2) `user_coins.balance` (described
as "placeholder, not yet wired"), and (3) a derived "DH credit equivalent"
computed from XP via `convertXPToDH()` — i.e. the wallet currently shows the
teen what their XP would buy in DH at a fixed conversion (10 XP = 1 DH). The
founder doc explicitly flags Coins as a deferred workstream and pins the
canonical shop URL to `/teen/wallet?tab=shop`.

Spending paths intended: shop reward redemption (cosmetics, partner discounts,
mystery boxes, exclusive access — see `shop_rewards.reward_type`), event
booking checkout (hybrid XP + DH cash through Stripe / CMI / Mobile Money),
and parent top-up flows. Parental ceilings are conceptually present
(`PARENTAL_APPROVAL_THRESHOLD_XP = 1000` triggers a parent gate on big XP
spends) but no per-month DH cap, no maximum top-up per period, and no
category whitelist exist anywhere in code or DB.

## 2. Code state

### 2.1 XP-to-DH ratio — defined in two places, diverging

- `lib/payments/xp-converter.ts:10` — `XP_TO_DH_RATE = 0.10` (i.e. **10 XP
  = 1 DH**). Used by hybrid payments, the wallet header, and the shop tab to
  show DH-equivalent prices. Backed by helpers `convertXPToDH`,
  `convertDHToXP`, `calculateHybridPayment` (lines 23-131), and thresholds
  `MIN_XP_FOR_PAYMENT = 50` (l.16), `PARENTAL_APPROVAL_THRESHOLD_XP = 1000`
  (l.13).
- `gamification-system/database/migrations/021_xp_payment_system.sql:69` —
  inserts `('xp_to_dh_rate', '100', 'XP required for 1 DH')` into
  `xp_payment_settings`. **This is 100 XP per DH, the inverse of the code
  constant.** DB live state confirms the row exists today with value `100`.
  No code path actually reads this setting (grep confirms zero readers); it
  is dead config that contradicts the canonical TS constant.

### 2.2 Parent top-up — `app/api/parent/topup/route.ts` (mock + broken write)

- Validates parent role, parent-teen link, and e-signature gate (l.9-69) —
  this part is wired.
- **At l.74-105 it `SELECT total_coins FROM profiles` then `UPDATE profiles
  SET total_coins = …`. The column `profiles.total_coins` does not exist in
  the live DB** (verified via `information_schema.columns`: profiles has only
  id, email, full_name, avatar_url, role, created_at, updated_at). Every
  top-up therefore silently fails the SELECT/UPDATE chain — or, more
  precisely, the SELECT errors with "column does not exist" and the route
  returns 404 "Teen non trouvé". No coins are ever credited to `user_coins`.
- It also inserts a row into `coin_transactions` (l.108-124) with
  `transaction_type: 'topup'`, `source_user_id: parentId`, `amount: totalCoins`
  — but the live count is **0 rows**, confirming the chain breaks before
  reaching the insert in production runs (or no parent has ever completed a
  top-up).
- The shop catalog (`/parent/topup/page.tsx:101-141`) is a hardcoded array of
  4 packs (100/250/500/1000 coins for 50/100/180/300 DH with bonus tiers).
  No DB-backed `topup_packages` table exists — packages cannot be A/B-tested
  or admin-edited without a code deploy.
- No Stripe / CMI / Mobile Money call is made for the DH side. `price` is
  passed in the body but never charged — this route credits coins *as if*
  the parent had already paid out-of-band. There is no real-money rail wired
  to this top-up flow today.

### 2.3 Hybrid payment route — `app/api/payments/hybrid/route.ts` (wired, but XP-only)

- Wires Zod validation (l.24-29), booking ownership check (l.81-93), parental
  approval branch when `xp >= 1000` (l.119-159), and uses the canonical
  `calculateHybridPayment` from xp-converter (l.106-110).
- XP debit at l.162-198: subtracts from `user_xp.total_xp`, inserts into
  `xp_transactions` with `type='purchase'`, updates booking. Then routes to
  Stripe (l.256-305), CMI (l.306-349, gated by feature flag), or Mobile Money
  (l.350-394, gated by feature flag).
- **This route only knows about XP and DH cash.** It never touches
  `user_coins.balance`. There is no code path anywhere that lets a teen
  spend Coins on a booking, and no path that converts XP → coins or
  coins → XP.

### 2.4 Reward shop debit — `gamification-system/features/shop/actions.ts`

- `purchaseReward()` at l.143-194 calls Postgres RPC `purchase_reward` with
  user id, reward id, optional promo. The RPC (per founder doc §6) atomically
  debits **`user_xp.total_xp`** and inserts into `shop_purchases`. No coin
  debit. No DH debit. Shop is XP-only.
- Catalog read (`getRewards`, l.60-91) goes through RPC `get_shop_rewards`
  off table `shop_rewards`. Live state: 26 rows across 9 reward_types
  (experience, physical_item, profile_customization, discount, mystery_box,
  exclusive_access, free_entry, digital_item, skip_queue) — i.e. the founder's
  intended spending paths exist as a catalog, but they are **all priced in
  XP**, not coins. There is no `currency` column on `shop_rewards`.

### 2.5 Wallet read path — `app/teen/wallet/page.tsx` and `wallet-hub-client.tsx`

- `app/teen/wallet/page.tsx:34` — explicitly hardcodes `coins: 0` with a
  TODO: "Fetch from user_coins table once it exists" — but the table **does
  exist** in DB (with 1 seeded row holding `balance: 100` for amine). The
  page never reads it.
- `app/teen/wallet/wallet-hub-client.tsx:88-92` — renders `walletData.coins`
  in a yellow pill, so the user always sees `0 coins` regardless of seeded
  reality.
- A second wallet read path exists at `app/api/teen/wallet/route.ts:21-25`
  which queries `user_coins` — but it filters by `eq('user_id', teenId)`
  while the actual column is **`teen_id`**. That query never returns rows
  even if balance exists. Same bug repeated for `user_xp` (l.28-32).
- `app/teen/coins/page.tsx` is a 12-line redirect to `/teen/wallet` (the
  legacy mock with totalCoins=1250 was deleted per the unifier pass).

## 4. Gaps (vision -> reality) — P0 / P1 / P2

### P0-1 — Top-up does not credit anything (catastrophic)
- **Vision**: parent pays in DH, coins land in teen's wallet.
- **Code**: `app/api/parent/topup/route.ts:74-97` reads/writes
  `profiles.total_coins`.
- **DB**: `profiles` has no `total_coins` column; SELECT errors out before
  INSERT. `coin_transactions` is empty. `user_coins` was never touched by
  any flow — its single row was seeded directly.
- **Action**: rewrite the top-up route to upsert `user_coins.balance` keyed
  on `teen_id`, increment `lifetime_earned`, then insert `coin_transactions`
  with the correct columns (`teen_id`, `source_id`, not `user_id`/
  `source_user_id`). Wire a real DH payment provider (Stripe/CMI) before
  crediting — currently a parent could call the route directly and "buy" coins
  for free if the column bug were fixed.

### P0-2 — XP-to-DH rate diverges 10x between code and DB
- **Vision**: a single canonical conversion rate.
- **Code**: `lib/payments/xp-converter.ts:10` says **10 XP = 1 DH**.
- **DB**: `xp_payment_settings.xp_to_dh_rate = 100` says **100 XP = 1 DH**.
- **Action**: pick one, delete the other. Either drop `xp_payment_settings`
  entirely (keep TS as SoT, simpler), or have `xp-converter.ts` read the
  setting at startup and cache it, and update the migration's seed to match.

### P0-3 — `parental_approvals` table missing -> hybrid pay 500s on big XP
- **Vision**: parent gate on large XP spends.
- **Code**: `app/api/payments/hybrid/route.ts:121-139` inserts into
  `parental_approvals`.
- **DB**: table does not exist.
- **Action**: ship the migration that creates the table + RLS, or remove
  the parental-gate branch.

### P0-4 — Wallet API uses wrong PK column on `user_xp`/`user_coins`
- **Code**: `app/api/teen/wallet/route.ts:24, 31` filter `eq('user_id',
  teenId)`.
- **DB**: column is `teen_id` in both tables.
- **Action**: change to `teen_id`. (Wallet UI dodges this bug because
  `app/teen/wallet/page.tsx` uses `getTeenDashboardData()` instead — but
  any external caller of the API gets {coins: 0, xp: 0} silently.)

### P1-5 — No XP -> Coins conversion exists anywhere
- **Vision (founder question)**: should teens convert XP -> coins, or only
  spend each in its own lane?
- **Code**: zero hits for `xp_to_coin`, `convert.*xp.*coin`. The two
  currencies are completely siloed today.
- **Action**: founder decision (see §6). If wanted, add an RPC
  `convert_xp_to_coins` with rate, ceiling, and ledger entries on both
  tables, plus a UI in the wallet.

### P1-6 — Coins are not a currency on `shop_rewards`
- **Vision**: cosmetics paid in coins, partner discounts in coins or hybrid.
- **DB**: 26 `shop_rewards` rows, all priced in `xp_cost` only, no
  `coin_cost` column.
- **Action**: add `coin_cost int` (nullable) and a `currency` enum, then
  classify the catalog (profile_customization rows are the natural coin-only
  candidates).

### P1-7 — Top-up packs are hardcoded in TSX
- **Code**: `app/parent/topup/page.tsx:101-141` defines 4 packs in JS.
- **Action**: move to a `topup_packages` table so admins can change pricing
  without redeploy and so analytics can attribute revenue per pack.

### P1-8 — No parental ceilings (max top-up per month, category limits)
- **Vision**: parental ceilings on monthly top-up and on what teens can spend
  on.
- **Code/DB**: zero hits for `monthly_cap | topup_limit | max_topup |
  ceiling`. No table, no constant, no UI.
- **Action**: introduce `parental_limits(parent_id, teen_id, max_monthly_dh,
  allowed_categories[])` and check it server-side in the topup route.

### P1-9 — `coin_transactions` ledger never written
- **DB**: 0 rows total. Even the seeded 100-coin balance has no ledger entry.
- **Action**: backfill seed transaction; gate any future direct
  `user_coins.balance` mutation behind a function that always writes the
  ledger row first.

### P2-10 — `family_subscriptions` shipped but unused (0 rows, no tier table)
- **DB**: table exists, no `subscription_tiers` / `subscription_packages` /
  `premium_subscriptions` table to back it.
- **Action**: either complete the subscription model or drop the empty table
  to reduce attack surface.

### P2-11 — `wallet-hub-client.tsx` always shows `coins: 0`
- **Code**: `app/teen/wallet/page.tsx:34` hardcodes `coins: 0` despite a
  live `user_coins.balance` of 100 for the seeded teen.
- **Action**: read `user_coins.balance` in `getTeenDashboardData()` and
  flow it through. Trivial change blocked only by the founder's
  "Coins not yet wired" stance in `docs/economy.md` §2.2.

## 5. Risks

- **Silent failure of top-up** (P0-1): parents will tap "Recharger" with no
  feedback that nothing was credited; no monitoring catches the SELECT
  error because it bubbles up as a generic 404.
- **Conversion drift** (P0-2): a 10x mismatch between code and DB rate is a
  ticking timebomb — first time someone "fixes" the divergence by wiring
  `xp_payment_settings` reads, every existing user's wallet display swings
  by 10x.
- **Currency confusion in UI**: the wallet header shows three pills (XP,
  Coins, DH credit) where DH credit is derived from XP, not from coins.
  Users will reasonably believe DH credit = coins, then be surprised when
  parent top-up "doesn't show up" in the DH column.
- **Missing audit trail**: zero rows in `coin_transactions` despite a
  non-zero `user_coins.balance` is exactly the situation Loi 09-08 / CNDP
  audits flag — financial data without a ledger.
- **Legal — Loi 09-08**: top-up route correctly gates on `e_signatures.terms_
  accepted` (good), but never logs the DH amount actually charged, never
  produces a receipt, and never offers refund/cancellation. For minors, this
  is a regulatory exposure the founder needs to address before launch.
- **No real DH rail**: the top-up route accepts a `price` in the body and
  ignores it. A bad actor with a parent account could mint unlimited coins
  for free if the column bug is patched without simultaneously wiring
  Stripe/CMI charge first.
- **Seeded coin balance with no ledger**: 100 coins exist for amine, no
  txn exists. If lifetime_spent goes up later via a (correct) shop debit
  but lifetime_earned stayed at 0, KPI dashboards will show "negative net
  earned" forever.
- **Double-spend window**: hybrid payment route subtracts XP via a non-
  atomic pattern (read `availableXP`, then `update total_xp = availableXP
  - amount` at l.165-170). Two concurrent payments on the same teen can
  both succeed and over-debit.

## 6. Open questions for the founder

1. **Canonical XP -> DH ratio**: 10 XP = 1 DH (TS constant) or 100 XP = 1 DH
   (DB seed)? They differ by 10x. Which one is the marketing message?
2. **Should teens be able to convert XP -> coins?** Or are XP and coins two
   strict lanes (XP for shop/bookings, coins for cosmetics)? Today neither
   direction exists.
3. **Should coins be spendable on event-booking checkout** (alongside XP +
   DH cash)? Today the hybrid route only knows XP + DH.
4. **Is the "DH credit equivalent" pill in the wallet header a feature or a
   bug?** It computes DH from XP, which can mislead a user who topped up
   coins and expected the DH pill to reflect coin balance.
5. **Per-month top-up cap per parent**: yes/no, what value (e.g. 500 DH/mo
   default, parent can raise)?
6. **Per-month spend cap per teen** and per category (event tickets vs
   profile cosmetics vs partner discounts)? Currently zero ceilings.
7. **Who pays for what**: should mystery_box, free_entry, exclusive_access,
   skip_queue stay XP-priced? Or are some of those coin-priced premium
   items?
8. **Real-money rail for top-up**: Stripe (international cards), CMI
   (Moroccan cards), Mobile Money (Orange Money / Inwi Money), or all
   three? Today the topup route charges nothing.
9. **What happens to coins on a refund**? E.g. parent disputes a top-up
   → must we be able to debit `user_coins.balance` retroactively even if
   the teen has spent some?
10. **Subscription tiers** — does Nivy plan a paid family subscription
    (Premium / Premium+) that includes monthly coin allowances? The empty
    `family_subscriptions` table suggests yes-but-paused.
