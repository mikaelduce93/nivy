# 05 — PSP automatic top-up activation runbook

> Wave Ops-D scaffolding. This runbook is the procedure to flip Nivy from
> manual top-up mode (parents transfer DH outside the app, an admin manually
> credits coins via `/admin/topups`) to automatic webhook mode (PSP webhooks at
> `/api/webhooks/{cashplus,wafacash,m2t}` credit coins directly).

## When to flip

The admin dashboard at `/admin/topups` shows a banner "Time to activate auto
mode" when **either** of these is true:

- **100 distinct families** have completed at least one successful top-up, OR
- **4 weeks** have elapsed since the first successful top-up.

The threshold is computed by the `manual_topup_threshold_status()` RPC
(`gamification-system/database/migrations/093_manual_topup_requests.sql`).

Do not flip the auto mode until **all** of the following are also true:

- [ ] PSP merchant contracts signed for the providers you want to enable
      (Cash Plus, Wafacash, M2T — at least one).
- [ ] PSP webhook secrets generated by the provider and added to Vercel env
      (see "Env vars" below).
- [ ] Webhook URL registered with the provider's dashboard:
      `https://nivy.app/api/webhooks/cashplus` (and same for `wafacash`, `m2t`).
- [ ] Provider's payload shape filled into the `TODO(founder)` stubs in
      `app/api/webhooks/<provider>/route.ts` so the parser pulls the right
      fields (`transaction_id`, `amount`, `customer_phone`, etc.).
- [ ] Parent-discovery logic confirmed — Nivy resolves the parent from the
      provider's payload via phone number (`profiles.phone`) or via an explicit
      `metadata.parent_id` we ask the provider to forward. If neither works
      reliably, the webhook will log "parent_teen_not_resolved" and a manual
      operator must reconcile.
- [ ] At least one end-to-end test in the provider's sandbox succeeded with
      `PSP_AUTO_TOPUP_ENABLED=true` set on a Vercel preview environment.

## Env vars to set

Set these in **Vercel project settings → Environment Variables → Production**:

| Variable | Value |
|---|---|
| `PSP_AUTO_TOPUP_ENABLED` | `true` (default `false` in manual mode) |
| `CASHPLUS_WEBHOOK_SECRET` | HMAC secret from Cash Plus dashboard |
| `WAFACASH_WEBHOOK_SECRET` | HMAC secret from Wafacash dashboard |
| `M2T_WEBHOOK_SECRET` | HMAC secret from M2T dashboard |

Do not commit secrets to the repo. After updating, redeploy to pick them up.

## Activation procedure

1. **Pre-flight on staging.** Set `PSP_AUTO_TOPUP_ENABLED=true` on the Vercel
   preview environment first. Use the provider's sandbox to send 3 test
   webhooks (one valid, one with bad signature, one duplicate). Confirm:
   - Valid → `payment_transactions` row written, `user_coins.balance` bumped.
   - Bad signature → 200 returned, no row written, log line "invalid signature".
   - Duplicate → 200 returned, `top_up_teen` returns `idempotent_replay=true`,
     no double-credit.

2. **Flip production.** Set `PSP_AUTO_TOPUP_ENABLED=true` in Vercel prod env,
   redeploy. The /admin/topups banner should disappear; the dashboard header
   chip flips from "Mode webhook auto: INACTIF" to "ACTIF".

3. **Monitor the first 10 transactions.** Run this SQL in the Supabase SQL
   editor and watch new rows arrive over the next hours:

   ```sql
   SELECT id, parent_id, teen_id, amount_dh, psp_provider, psp_reference,
          status, created_at, succeeded_at
   FROM payment_transactions
   WHERE psp_provider IN ('cashplus','wafacash','m2t')
   ORDER BY created_at DESC
   LIMIT 20;
   ```

   For each, verify:
   - `status = 'succeeded'`
   - `escrow_ledger` has a paired row with `direction='top_up'` and
     matching `related_payment_id`.
   - `coin_transactions` has a paired `transaction_type='topup'` row.
   - `user_coins.balance` for the teen incremented by `amount_dh * 100`.

4. **Watch the audit logs** for any `psp.webhook.received` rows that didn't
   produce a `payment_transactions` row (means parent resolution failed):

   ```sql
   SELECT created_at, target_type, target_id, payload
   FROM admin_audit_logs
   WHERE action = 'psp.webhook.received'
     AND (payload->>'auto_credit_enabled')::boolean = true
     AND (payload->'resolved_link') IS NULL
   ORDER BY created_at DESC
   LIMIT 50;
   ```

   Each such row needs manual reconciliation via the manual-topups UI.

## Rollback

If anything goes wrong: set `PSP_AUTO_TOPUP_ENABLED=false` in Vercel prod env
and redeploy. Webhooks will continue to verify signatures and log events but
will no longer credit coins. Parents revert to filing manual requests via
`/parent/topup/manual`. Already-credited transactions are not affected — the
RPC is idempotent and prior `payment_transactions` rows remain `succeeded`.
