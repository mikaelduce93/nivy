## 1. View Transitions API — TICKET-024

### Provider exists and is mounted
- File: `components/providers/view-transitions-provider.tsx` (148 lines).
- Mounted in `app/layout.tsx` line 18 (import) and lines 228 / 293 (JSX), wrapping `AppProviders` inside `PerformanceProvider`. Nesting is correct: VT lives below the perf observer (so its work is timed) but above every feature surface.
- Strategy: capture-phase delegated `click` listener on `document`, walks the DOM up to the closest `<a>`, validates same-origin / no-modifier / non-`_blank` / non-download / no `data-view-transition="off"` opt-out, then `event.preventDefault()` + `document.startViewTransition(() => router.push(...))`.

### Five view-transition-name pairs confirmed
Searched repo for `vt-(quest|mentor|listing|restaurant|feed)`. All five surfaces have **both** a list-side caller and a detail-side hero:

| # | Name prefix          | List page                                            | Detail page                                                        |
|---|----------------------|------------------------------------------------------|--------------------------------------------------------------------|
| 1 | `vt-quest-<id>`      | `app/teen/quests/quests-hub-client.tsx`              | `app/teen/quests/[id]/quest-detail-client.tsx`                     |
| 2 | `vt-mentor-<id>`     | `app/teen/mentors/page.tsx`                          | `app/teen/mentors/[id]/page.tsx`                                   |
| 3 | `vt-listing-<id>`    | `app/marketplace/page.tsx`                           | `app/marketplace/listings/[id]/page.tsx`                           |
| 4 | `vt-restaurant-<id>` | `app/teen/food/page.tsx` (via `defi-card`-style ref) | `app/teen/food/[partner_id]/page.tsx`                              |
| 5 | `vt-feed-<id>`       | `app/teen/feed/feed-list.tsx`                        | `app/teen/feed/[id]/page.tsx`                                      |

The allowlist is also self-documented in `app/globals.css` lines 1837–1844, which is the right defensive comment — adding a 6th surface forces the engineer to update both code and CSS.

### prefers-reduced-motion — two-layer skip confirmed
1. **JS layer** (`view-transitions-provider.tsx` lines 50–55, 124): `prefersReducedMotion()` returns `matchMedia('(prefers-reduced-motion: reduce)').matches`; when true, the click handler `return`s early before calling `startViewTransition`, so the browser never even snapshots the DOM. The user gets a vanilla `<a>` navigation.
2. **CSS layer** (`globals.css` lines 1851–1856): `@media (prefers-reduced-motion: reduce) { ::view-transition-group(*), ::view-transition-old(*), ::view-transition-new(*) { animation: none !important; } }` — defence-in-depth for any callsite that bypasses the provider.

Default crossfade timing: `280ms` cubic-bezier(0.23, 1, 0.32, 1) (`globals.css` 1821–1825). Per-surface morph: `320ms` cubic-bezier(0.16, 1, 0.3, 1) (1832–1835). Both match `PAGE_TRANSITION_EASINGS.smooth` for visual consistency with framer-motion-driven page transitions.

**Score: 5 / 5 — exemplary implementation. The only nit is that the `data-view-transition="off"` opt-out is undocumented outside the provider source — worth a one-line callout in component docs.**

## 2. CSS `color-mix()` — TICKET-012

**Status: NOT IMPLEMENTED as specified.** The `*-soft` tokens exist but are defined as **direct OKLCH literals**, not via `color-mix(in oklch, ...)`.

### What was found in `app/globals.css`
- `:root` (lines 232–236) defines four soft tokens:
  ```css
  --brand-soft:   oklch(0.75 0.12 290);   /* light mode */
  --accent-soft:  oklch(0.72 0.14 25);
  --success-soft: oklch(0.80 0.12 165);
  --info-soft:    oklch(0.78 0.12 230);
  ```
- `.dark` (lines 360–364) mirrors the same four tokens with brighter L/C values.
- `@theme inline` (lines 538–541) re-exports them as `--color-brand-soft` etc. so Tailwind can produce `bg-brand-soft`, `border-success-soft`, `ring-info-soft`.

### Three deltas vs. the ticket spec
1. **No `color-mix()` calls anywhere in `globals.css`.** Grep returned 0 matches. The comment block on `card.tsx` line 9–10 says "backed by the color-mix()-derived *-soft tokens added in TICKET-012" — that documentation is now stale; the tokens are static OKLCH values, not mathematically derived at render time.
2. **`--warning-soft` is missing.** No definition in `:root` or `.dark`, no entry in `@theme inline`. Searched `--warning-soft|warning-soft` — only hit is in `components/ui/card.tsx` (likely an unmounted variant referencing an undefined token; Tailwind will emit `bg-warning-soft` → `var(--color-warning-soft)` → unset, falling back to inherited bg).
3. **`--danger-soft` is missing.** Same situation. The destructive semantic exists (`--destructive`) but no soft companion.

### Why this matters
The intent of `color-mix()` was to derive soft tokens from their semantic source automatically (`color-mix(in oklch, var(--primary) 18%, var(--background))`), so a future palette swap on `--primary` would propagate without re-tuning `--brand-soft`. Hard-coding OKLCH values instead means tokens drift on palette changes. Functionally the surfaces look correct *today*, so this is technical debt, not a regression.

**Score: 2 / 5 — tokens are present and used, but the `color-mix()` derivation is absent and 2 of the 6 expected tokens (warning-soft, danger-soft) are missing.**

## 3. Container queries

### Adoption count
- **Tailwind classes** (`@container/<name>`) in `.tsx`: **2 files**.
  - `components/ui/card.tsx:141` — `'@container/card-header grid auto-rows-min ...'`
  - `components/ui/field.tsx:52` — `'group/field-group @container/field-group flex w-full ...'`
- **Raw CSS `@container` rules**: **0** in app code. The 4 `@container` matches are 2 in component .tsx (above), 2 in `docs/vision/audit-ui-ux-2026/{TICKETS,MASTER_UX_AUDIT}.md` (planning docs).
- **`container-type:` declarations in CSS**: **0** in app code.

### Reading
The two component-level `@container/<name>` declarations declare a containment context, but no descendant uses container query variants (`@sm:`, `@md:`) — so today they are inert *naming* without any responsive readers. Tailwind v4 wires `container-type: inline-size` automatically when it sees `@container/<name>` in a class list, so the contexts are valid; they just aren't queried.

This is a "scaffolding present, not yet leveraged" situation. The Wave 1–3 build did not need container queries to ship; the responsive grid is media-query-based.

**Score: 2 / 5 — Tailwind v4 wiring is correct, but adoption is functionally zero. Backlog: convert 5–10 high-value cards (quest cards, mentor cards, feed posts) to `@container/card`-based responsive layouts.**

## 5. CSS `@layer` ordering

`globals.css` defines four `@layer` blocks:

| Line | Block             | Purpose                                                |
|------|-------------------|--------------------------------------------------------|
| 458  | `@theme inline`   | Tailwind v4 design-system mapping (not technically a layer, but ordering-equivalent in Tailwind v4) |
| 550  | `@layer base`     | element resets, focus-visible, selection styling, typography |
| 627  | `@layer utilities`| text gradients, glow effects                           |
| 938  | `@layer utilities`| (continuation — second utility block)                  |
| 1186 | `@layer utilities`| (third utility block)                                  |
| 1318 | `@layer utilities`| (fourth utility block)                                 |

**Explicit ordering preamble (`@layer reset, base, components, utilities;`) is NOT declared.** Tailwind v4 (`@import "tailwindcss"` line 1) injects its own ordering implicitly, which is the canonical pattern in v4 — you do not normally hand-roll a preamble. Cascade order is therefore: Tailwind preflight → `@theme inline` → `@layer base` (custom) → `@layer utilities` (custom) → unlayered (none in this file).

There is **no custom `@layer components`** — components rely on Tailwind classes at the call site rather than CSS-layer-bound classes, which is correct for a v4 utility-first codebase.

**Score: 4 / 5 — ordering is correct via Tailwind v4 conventions; one half-mark deduction because the four separate `@layer utilities` blocks (lines 627, 938, 1186, 1318) could be consolidated for readability. No functional issue.**

| `canvas-confetti`    | 1.9.4   | ~9 KB            | On-demand                      | Imported only inside celebration components (XP-up, quest-complete). Should be confirmed dynamic-imported per surface. |

### lucide-react import-shape audit
- Searched `from "lucide-react"` across the codebase: **252 occurrences in 250 files** — every single import uses the named-import shape `import { ChevronRight, Sparkles, ... } from 'lucide-react'`.
- Searched `import * as ... from 'lucide-react'`: **0 matches** in app code (the one match in `components/teen/defi-card.tsx` is a misread by the regex; on inspection the file uses named imports too).

Combined with `optimizePackageImports: ['lucide-react']` in `next.config.mjs`, this means Webpack/Turbopack rewrites every `lucide-react` import into a single-icon module — no full-icon-set barrel ships to the client. **Tree-shakability: confirmed.**

### Recommendations
1. Run `npm run analyze` (already wired via `@next/bundle-analyzer` in `next.config.mjs`) before launch to capture per-route JS weight. Goal: every public route under 180 KB transferred.
2. Audit `framer-motion` — consider replacing 1–2 simple uses (e.g. fade-in transitions) with CSS `@starting-style` + `transition` to shed motion-library weight from the homepage.
3. Confirm `canvas-confetti` is dynamic-imported at *every* call site (worth a follow-up search if not already done in W3-A11).
4. Once PPR ships (see §8), the marketing pages will pre-render their static shell and only stream the dynamic gamification widgets — biggest single TTFB win available.

**Score: 4 / 5 — lucide tree-shaking is correct, lazy boundaries land in the right places, but framer-motion remains a sizeable always-loaded dep. Bundle analyzer should be run pre-launch.**
